## 1. Введение

### 1.1 Контекст и проблематика

Детейлинг-студии — это автомастерские, специализирующиеся на детальной химической чистке, полировке, защитных покрытиях и реставрации автомобилей. В отличие от обычных автомоек, услуги детейлинга:

- Дорогие и материалоёмкие (полироли, абразивы, защитные составы стоят значительно).
- Длительные (от 2 часов до нескольких суток на одно авто).
- Требуют квалифицированного мастера, привязанного к конкретному автомобилю на всё время работы.
- Зависят от типа кузова, состояния авто, выбранного состава работ.

Типичные операционные проблемы небольшой детейлинг-студии:

1. **Неконтролируемый расход материалов.** Мастер не учитывает, сколько граммов/миллилитров химии он реально израсходовал; владелец видит только закупки и итоговую выручку, а маржа размыта.
2. **Запись «на коленке».** Клиенты записываются через мессенджеры и звонки; админ ведёт расписание в блокноте или Google Календаре. Конфликты слотов, потерянные записи, нет напоминаний — обычное дело.
3. **Слепое планирование закупок.** Без статистики расхода не знают, когда и сколько заказывать у поставщика. Заказы идут «по ощущениям», бывают простои из-за отсутствия материалов.
4. **Нет прозрачности перед клиентом.** Клиент не видит фото «до/после», не получает структурированных уведомлений о статусе работ.

### 1.2 Видение продукта

**One-liner:** «1С + YClients для детейлинг-студии: один SaaS для записи клиентов и учёта материалов с прицелом на дальнейшую аналитику маржинальности».

**Расширенное видение:** SaaS-платформа, объединяющая в одном продукте:

- Онлайн-запись клиентов (как в YClients/Dikidi).
- Складской учёт материалов с привязкой к выполненным услугам (как в 1С:Розница, но проще).
- Рабочее место мастера с фото-фиксацией и контролем фактического расхода.
- Аналитику маржинальности по услугам и мастерам (Phase 2).

**Принципиальные ограничения MVP:**

- Один тенант = одна студия. Мультитенантность отложена в Phase 3, но архитектурно не блокируется.
- Поддерживается несколько филиалов в рамках одной студии (один владелец, один прайс, общий справочник мастеров и материалов, склад на каждый филиал свой).
- Только русский язык, валюта RUB, часовой пояс — выбирается на филиал из списка.

### 1.3 Цели MVP

| № | Цель | Критерий достижения |
|---|------|---------------------|
| G1 | Дать студии возможность вести запись клиентов онлайн без сторонних сервисов | Клиент может записаться через client-приложение без вмешательства администратора, мастер видит назначенную запись в master-приложении |
| G2 | Обеспечить фактический учёт расхода материалов по каждому наряду | По каждому завершённому наряду в системе зафиксированы списанные SKU, количества и причина расхождения с нормой (если есть) |
| G3 | Дать владельцу прозрачность по остаткам, истории движений и закупкам | В админке доступны отчёты «остатки на дату», «движения за период», «закупки у поставщика», «низкие остатки» |
| G4 | Обеспечить чистую DDD-архитектуру под будущее развитие | Доменные модели не зависят от ORM, агрегаты публикуют события, контексты имеют явные границы и ACL между собой |
| G5 | Запуск всей системы одной командой `docker-compose up` | Локальное развёртывание занимает < 5 минут от `git clone` до работающей системы |

### 1.4 Ключевые продуктовые метрики (KPI)

KPI для пилотной студии после внедрения MVP:

| Метрика | Цель |
|---------|------|
| Доля записей, созданных клиентом онлайн (не админом вручную) | ≥ 40% |
| Доля нарядов с зафиксированным фактическим расходом материалов | ≥ 90% |
| Доля нарядов с фото «до/после» | ≥ 95% |
| Среднее время от поступления заявки клиента до подтверждения | ≤ 15 минут |
| No-show rate (клиент не пришёл без отмены) | ≤ 10% |
| Среднее отклонение фактического расхода от нормы | ≤ 15% |

Технические KPI:

| Метрика | Цель |
|---------|------|
| API latency P95 (читающие запросы) | ≤ 300 ms |
| API latency P95 (записывающие запросы) | ≤ 600 ms |
| Frontend FCP (Largest Contentful Paint) | ≤ 2.5 s на 4G |
| Доступность сервиса (uptime) | ≥ 99% (best-effort на MVP) |
| Покрытие unit-тестами доменного слоя | ≥ 85% |

### 1.5 Не цели (вынесено за скоуп MVP)

- Мобильные нативные приложения (iOS/Android). Только Web и PWA.
- Полноценный биллинг подписки SaaS — только демонстрационная заглушка.
- Маркетплейс материалов и интеграции с поставщиками для авто-заказа.
- AI-распознавание авто/повреждений по фото.
- Интеграции с 1С, МойСклад и другими ERP.
- Полная мультитенантность с супер-админом платформы.
- Программа лояльности, скидочные карты, абонементы.
- Внутренние мессенджеры между сотрудниками.
- Финансовый/бухгалтерский учёт (касса, расчётный счёт, ЗП).

## 2. Глоссарий (Ubiquitous Language)

> Глоссарий используется как доменный словарь во всём коде, БД, API и UI. Добавление нового термина требует обоснования и согласования.

### 2.1 Организационные сущности

| Термин | Определение |
|--------|-------------|
| **Студия** (Studio) | Юридическое лицо или ИП, владеющее одним или несколькими филиалами. В MVP — один экземпляр на инстанс. |
| **Филиал** (Branch) | Физическая точка обслуживания клиентов: адрес, рабочие часы, свой склад, свой набор боксов. |
| **Бокс** (Bay) | Рабочее место в филиале (один пост). Одновременно в боксе обслуживается одно авто. |
| **Владелец** (Owner) | Пользователь с максимальными правами, видит финансы, себестоимость, аналитику. |
| **Администратор-приёмщик** (Manager) | Сотрудник, ведущий запись клиентов, приёмку, склад без цен закупки. |
| **Мастер** (Master) | Сотрудник, выполняющий услуги, фиксирующий фактический расход материалов и фото. |
| **Клиент** (Client) | Физическое лицо, заказывающее услуги. Может быть зарегистрированным или гостем. |

### 2.2 Каталог и услуги

| Термин | Определение |
|--------|-------------|
| **Категория услуг** (ServiceCategory) | Группировка услуг для UI: «Мойка», «Полировка», «Химчистка», «Защитные покрытия». |
| **Услуга** (Service) | Самостоятельная единица прайса с названием, описанием, длительностью и ценой. |
| **Тариф услуги** (ServicePricing) | Цена услуги в зависимости от типа кузова авто (седан / кроссовер / внедорожник / минивэн). |
| **Норма расхода** (MaterialNorm) | Связь «услуга → SKU → количество», задающая ожидаемый расход материала на единицу услуги. |
| **Тип кузова** (BodyType) | Перечислимое значение: `SEDAN`, `HATCHBACK`, `CROSSOVER`, `SUV`, `MINIVAN`, `PICKUP`, `COUPE`, `OTHER`. |

### 2.3 Учёт материалов

| Термин | Определение |
|--------|-------------|
| **Номенклатура / SKU** (Sku) | Карточка позиции склада: название, артикул, ед. изм., штрих-код, минимальный остаток. |
| **Группа номенклатуры** (SkuGroup) | Классификация SKU: «Химия», «Расходники», «Инструмент». |
| **Единица измерения** (UnitOfMeasure) | Базовая (мл, г, шт., м) и упаковочная (канистра 5 л, пачка 100 шт.); хранится коэффициент пересчёта. |
| **Поставщик** (Supplier) | Контрагент, у которого закупаются материалы. Карточка содержит контакты, банковские реквизиты, историю поставок. |
| **Партия** (Batch / Lot) | Поступившая партия SKU с фиксированной ценой закупки и сроком годности. Учёт остатков ведётся по партиям. |
| **Складское движение** (StockMovement) | Атомарная операция изменения остатка: приход, списание, корректировка, перемещение, инвентаризация. |
| **Приход** (Receipt) | Движение «+» по партии, оформляется приходным ордером со ссылкой на накладную поставщика. |
| **Списание на наряд** (ConsumptionWriteOff) | Движение «−» по партии, привязано к закрытому наряду и конкретному SKU. |
| **Корректировка** (Adjustment) | Ручное движение «+/−» с обязательным комментарием и уровнем подтверждения. |
| **Перемещение** (Transfer) | Парное движение между складами разных филиалов. |
| **Инвентаризация** (StockTaking) | Сверка фактических остатков с системными; результат — массовая корректировка с ревизионным документом. |
| **Себестоимость** (Cost) | Цена единицы SKU, используемая для расчёта стоимости списания. Метод — средневзвешенная. |
| **Минимальный остаток** (ReorderLevel) | Порог по SKU и филиалу, при достижении которого создаётся уведомление о низком остатке. |

### 2.4 Клиенты и автомобили

| Термин | Определение |
|--------|-------------|
| **Клиент** (Client) | Контрагент-физлицо: ФИО, телефон, email, согласие на ПДн, источник, сегмент. |
| **Автомобиль клиента** (Vehicle) | Принадлежащее клиенту авто: марка, модель, тип кузова, госномер, VIN, год, цвет. |
| **Гость** (Guest) | Клиент, записавшийся без полноценной регистрации (только телефон + одноразовый OTP). |

### 2.5 Запись и расписание

| Термин | Определение |
|--------|-------------|
| **График работы** (WorkSchedule) | Календарь рабочих часов филиала (по дням недели и исключениям). |
| **График мастера** (MasterSchedule) | Рабочие часы конкретного мастера, могут отличаться от графика филиала. |
| **Слот** (Slot) | Единица времени в шаге 15 минут на конкретного мастера в конкретном боксе. |
| **Запись** (Appointment) | Заявка клиента на услугу/услуги в конкретное время к конкретному мастеру. |
| **Состояния записи** | `PENDING_CONFIRMATION` → `CONFIRMED` → `IN_PROGRESS` → `COMPLETED` / `CANCELLED` / `NO_SHOW`. |
| **Окно отмены** (CancellationWindow) | Период до начала записи, в который клиент может отменить запись самостоятельно (по умолчанию 24 ч). |
| **Запрос на отмену** (CancellationRequest) | Заявка клиента на отмену внутри окна согласования; требует подтверждения админа. |

### 2.6 Наряды (выполнение работ)

| Термин | Определение |
|--------|-------------|
| **Наряд** (WorkOrder) | Документ выполнения услуги. Создаётся при переходе записи в статус `IN_PROGRESS`. |
| **Состояния наряда** | `OPEN` → `IN_PROGRESS` → `AWAITING_REVIEW` → `CLOSED` / `CANCELLED`. |
| **Фото «до»** (BeforePhoto) | Фотография авто в начале работ; обязательно ≥ 1 фото для закрытия наряда. |
| **Фото «после»** (AfterPhoto) | Фотография авто после выполнения работ; обязательно ≥ 1 фото для закрытия наряда. |
| **Факт расхода** (ActualConsumption) | Запись о реально израсходованном SKU и количестве в рамках наряда. |
| **Расхождение нормы** (NormDeviation) | Разница между нормативным и фактическим расходом. Если превышает порог (по умолчанию 15%), мастер обязан указать причину. |

### 2.7 Уведомления

| Термин | Определение |
|--------|-------------|
| **Канал уведомлений** (NotificationChannel) | `EMAIL`, `SMS`, `TELEGRAM`, `PUSH` (PWA). |
| **Шаблон уведомления** (NotificationTemplate) | Параметризованный текст с подстановкой данных события. |
| **Подписка пользователя** (NotificationSubscription) | Какие события и через какие каналы получает пользователь. |

### 2.8 Биллинг (демо-заглушка)

| Термин | Определение |
|--------|-------------|
| **Тариф** (Plan) | Тарифный план SaaS: название, цена/мес, лимиты (число филиалов, мастеров, записей в месяц). |
| **Подписка** (Subscription) | Текущий план студии: дата начала, дата следующего списания, статус (`TRIAL`, `ACTIVE`, `PAST_DUE`, `CANCELLED`). |

## 3. Целевая аудитория и роли

### 3.1 Персонажи (Personas)

#### Персонаж 1: Андрей, владелец небольшой студии

- 32 года, бывший мастер, 3 года назад открыл свою студию.
- 1 филиал, 3 бокса, 4 мастера в команде.
- Ведёт расписание в Telegram-чате со студией; склад в Excel-таблице, которую обновляет раз в неделю.
- Боль: не понимает, какие услуги маржинальны; подозревает, что мастера «выливают» лишнюю химию.
- Хочет: единое место для всего, цифры по марже, контроль расхода.

#### Персонаж 2: Ольга, администратор-приёмщик

- 26 лет, работает в студии Андрея, отвечает на звонки и ведёт запись.
- 12 часов в смену, обрабатывает 30–50 обращений в день через Instagram, Telegram, звонки.
- Боль: ручная синхронизация расписания между мессенджерами и календарём, ошибки в записях, конфликты слотов.
- Хочет: единый канал записи, автоматические напоминания клиентам, простой интерфейс.

#### Персонаж 3: Дмитрий, мастер-полировщик

- 34 года, специалист по полировке и защитным покрытиям.
- Работает с конкретным авто 4–8 часов, делает фото «до/после» в личный телефон, отправляет в чат с клиентом.
- Боль: помнить, какие материалы и сколько использовал, отчитываться вручную; путаница с фото.
- Хочет: открывать наряд на телефоне, нажать «начал/закончил», быстро добавить фото и расход.

#### Персонаж 4: Иван, клиент

- 38 лет, владелец BMW X5, требователен к качеству.
- Записывается раз в 2–3 месяца, ценит время.
- Боль: висеть на телефоне, чтобы записаться; не получать подтверждения; не видеть, что именно делали с авто.
- Хочет: записаться за 30 секунд с телефона, получить смс-подтверждение, увидеть фото «до/после».

### 3.2 Роли в системе

| Роль | Код | Назначение | Назначается |
|------|-----|------------|-------------|
| **OWNER** | `OWNER` | Владелец студии, видит всё | Один на инстанс, при первичной инициализации |
| **MANAGER** | `MANAGER` | Администратор-приёмщик | Владельцем |
| **MASTER** | `MASTER` | Мастер | Владельцем или менеджером |
| **CLIENT** | `CLIENT` | Клиент студии | Самостоятельной регистрацией или авто-созданием при гостевой записи |

> **Решение по «сотрудникам».** В MVP отдельной роли «кладовщик» нет; складские операции (приход, инвентаризация, корректировки, поставщики) выполняет роль `MANAGER`. Один пользователь может иметь **несколько ролей одновременно** (например, Андрей-владелец одновременно `OWNER` + `MASTER` для собственных записей).

### 3.3 Матрица доступа (RBAC)

> Обозначения: ✓ — полный доступ, R — только чтение, ⊘ — нет доступа, * — особые правила (см. примечание).

#### 3.3.1 Идентификация и пользователи

| Раздел / Действие | OWNER | MANAGER | MASTER | CLIENT |
|---|:-:|:-:|:-:|:-:|
| Просмотр своего профиля | ✓ | ✓ | ✓ | ✓ |
| Редактирование своего профиля | ✓ | ✓ | ✓ | ✓ |
| Просмотр списка сотрудников | ✓ | ✓ | R* | ⊘ |
| Создание сотрудника (приглашение) | ✓ | ✓ | ⊘ | ⊘ |
| Изменение ролей | ✓ | ⊘ | ⊘ | ⊘ |
| Деактивация сотрудника | ✓ | ⊘ | ⊘ | ⊘ |
| 2FA для своего аккаунта | ✓ | ✓ | ✓ | ✓ |

\* Мастер видит только список других мастеров (для перевода записи), без email и контактов.

#### 3.3.2 Каталог услуг

| Раздел / Действие | OWNER | MANAGER | MASTER | CLIENT |
|---|:-:|:-:|:-:|:-:|
| Просмотр списка услуг | ✓ | ✓ | ✓ | R |
| Создание услуги | ✓ | ⊘ | ⊘ | ⊘ |
| Редактирование услуги | ✓ | ⊘ | ⊘ | ⊘ |
| Изменение цены | ✓ | ⊘ | ⊘ | ⊘ |
| Активация/деактивация | ✓ | ⊘ | ⊘ | ⊘ |
| Привязка норм расхода | ✓ | ⊘ | ⊘ | ⊘ |

> **Решение.** Управление прайсом и нормами расхода — только владелец. Менеджер не может менять цены, чтобы избежать злоупотреблений.

## 4. Сценарии пользователей (User Journeys)

> Жирным выделены ключевые точки взаимодействия с системой. В скобках — ссылка на функциональное требование (см. раздел 5).

### 4.1 Первичная запись клиента онлайн (happy path)

**Актор:** Клиент Иван, мобильный браузер.

1. Иван заходит на `client.detailing.example` (PWA-хост студии) → видит лендинг с CTA «Записаться».
2. Жмёт «Записаться» → попадает в визард онлайн-записи (FR-5.5.2).
3. **Шаг 1.** Выбирает услуги из каталога (FR-5.2.4): «Полировка кузова» + «Защитное покрытие».
4. **Шаг 2.** Указывает свой авто: марка «BMW», модель «X5», тип кузова «Внедорожник» (тип определяет цену по тарифу услуги — FR-5.2.6). Госномер и VIN — опционально.
5. **Шаг 3.** Видит каталог свободных слотов на ближайшие 14 дней (FR-5.5.4). Выбирает удобную дату/время. Может выбрать конкретного мастера или «любой свободный».
6. **Шаг 4.** Указывает контакт: телефон.
7. **Подтверждение OTP.** Получает SMS-код на номер, вводит → авто-создаётся карточка клиента (если не существовала) — FR-5.1.4.
8. **Запись создана** в статусе `CONFIRMED` (правило: онлайн-запись клиента подтверждается автоматически — FR-5.5.5).
9. **Уведомление.** Получает SMS «Вы записаны: дата, время, услуги, адрес студии, ссылка на отмену» (FR-5.7.3).
10. **За 24 часа.** Получает напоминание SMS+Telegram (если привязан) с кнопкой «Подтвердить, что приду» (FR-5.7.4).
11. **В день записи.** Открывает client-приложение, видит экран «До приёма N часов», адрес, контакт мастера.
12. Прибывает на филиал → менеджер встречает, наряд открывается (см. сценарий 4.4).

### 4.2 Перенос/отмена записи клиентом

**Сценарий 4.2.1: отмена за >24 ч (самостоятельная).**

1. Клиент в client-приложении в разделе «Мои записи» нажимает «Отменить» на будущей записи.
2. Система проверяет: до начала ≥ 24 часов — разрешено (FR-5.5.7).
3. Клиент указывает причину (опционально: «Поменялись планы», «Не нужно», «Найду другой сервис», «Свободный текст»).
4. Запись переходит в статус `CANCELLED`. Слот освобождается.
5. Менеджеру приходит уведомление об отмене (FR-5.7.5).

**Сценарий 4.2.2: отмена за <24 ч (через согласование).**

1. Клиент жмёт «Отменить» — система видит, что осталось < 24 часов.
2. Открывается форма «Заявка на отмену»: причина обязательна.
3. Создаётся `CancellationRequest`, запись остаётся в статусе `CONFIRMED`, но с флагом «есть запрос на отмену».
4. Менеджер получает уведомление (Telegram/email), видит карточку запроса, может «Подтвердить отмену» или «Отклонить».
5. При подтверждении — запись отменяется, клиент уведомляется.
6. При отклонении — запись остаётся, клиент уведомляется с причиной отклонения.

**Сценарий 4.2.3: перенос на другое время.**

1. Клиент жмёт «Перенести» → попадает в шаг выбора времени (как в первичной записи).
2. Если до начала ≥ 24 ч — перенос автоматический.
3. Если < 24 ч — заявка на перенос на согласование менеджеру (аналогично 4.2.2).
4. После переноса слот старого времени освобождается, новый — занимается.

### 4.3 Гостевая запись

**Актор:** Клиент Сергей, не хочет создавать аккаунт.

1. На лендинге выбирает «Записаться без регистрации».
2. Проходит шаги выбора услуг и авто (4.1.3–4.1.5).
3. Указывает: ФИО, телефон, email (опционально).
4. Подтверждает OTP по SMS — это и есть «авторизация на одну запись».
5. Запись создана, в системе создаётся «лёгкая» карточка клиента с пометкой `GUEST`.
6. После записи система предлагает: «Хотите получать уведомления и видеть историю в личном кабинете? Создайте пароль / подключите Яндекс / Telegram». Если соглашается — карточка переходит в полноценный аккаунт.

### 4.4 Рабочий день мастера

**Актор:** Дмитрий, master-приложение на планшете в боксе.

1. Утром открывает master-приложение → видит ленту записей на сегодня (FR-5.5.8).
2. Прибывает первый клиент. Менеджер создаёт «Открыть наряд» из карточки записи (или мастер сам жмёт «Принять авто» в master-приложении).
3. **Открытие наряда** (FR-5.6.2). Мастер фотографирует авто «до» (≥ 1 фото, обязательно), отмечает повреждения (опционально, текстом или маркерами на схеме — Phase 2).
4. Запись переходит в `IN_PROGRESS`, наряд — в `OPEN`.
5. Мастер выполняет работу. По мере списания материалов — нажимает «Добавить расход» в наряде, выбирает SKU, вводит количество (FR-5.6.4). Норма расхода (если есть) подсказывается.
6. Закончив, фотографирует авто «после» (≥ 1 фото).
7. **Закрытие наряда** (FR-5.6.5). Если фактический расход отклоняется от нормы более чем на 15% — система требует обязательный комментарий. Мастер указывает причину («Сильное загрязнение», «Использовал больше состава из-за состояния ЛКП» и т.п.).
8. Наряд переходит в `AWAITING_REVIEW` → автоматически в `CLOSED` (если нет настройки «обязательное ревью владельцем»).
9. **Списание материалов.** Система автоматически создаёт `StockMovement` типа `CONSUMPTION_WRITE_OFF` по каждой строке факта расхода (FR-5.3.6), с привязкой к партиям по правилу FEFO (First Expired First Out, или FIFO, если срока годности нет).
10. **Уведомление клиенту.** Клиент получает уведомление «Работы завершены» с фото «до/после» (FR-5.7.6).
11. Мастер переходит к следующей записи.

### 4.5 Приёмка нового товара (приход)

**Актор:** Менеджер Ольга, админ-приложение.

1. Привезли товар от поставщика «АвтоХимия Плюс».
2. Ольга идёт в раздел «Склад → Приходы → +Создать».
3. Выбирает поставщика из справочника (или создаёт нового на лету).
4. Указывает номер и дату накладной поставщика, прикрепляет PDF/фото (опционально).
5. Добавляет позиции:
   - SKU «Полироль X 1L» (если нет — создаёт), кол-во 5, цена 2 800 ₽/шт., срок годности 12.2026.
   - SKU «Микрофибра 40×40», кол-во 50, цена 90 ₽/шт.
6. Жмёт «Провести» → создаются партии (две новые партии для двух SKU), создаются `StockMovement` типа `RECEIPT`, обновляются остатки и средневзвешенная себестоимость (FR-5.3.5).
7. История прихода доступна в карточке поставщика и в общем журнале движений.

### 4.6 Контроль остатков и низкого запаса

**Актор:** Владелец Андрей.

1. Утром получает Telegram-уведомление от бота: «Низкий остаток: 3 SKU (Полироль Y, Шампунь Z, Микрофибра 40×40)».
2. Открывает админ-приложение, раздел «Склад → Низкие остатки» — видит список с прогнозом «закончится через ~ N дней» (на основе среднего расхода — Phase 2, в MVP только текущий остаток vs минимум).
3. Делает заявку поставщику (вне системы, в чате/звонком), при поступлении товара — сценарий 4.5.

### 4.7 Инвентаризация филиала

**Актор:** Менеджер Ольга.

1. Раз в месяц студия проводит инвентаризацию.
2. В админ-приложении: «Склад → Инвентаризация → Создать инвентаризацию».
3. Указывает филиал, дату/время. Система фиксирует «системные остатки на этот момент» по всем SKU.
4. Печатает инвентаризационную ведомость (PDF) или работает в приложении.
5. Обходит склад, в каждой строке вводит фактическое количество. Система рассчитывает расхождение.
6. По окончании жмёт «Провести инвентаризацию» → создаются `StockMovement` типа `ADJUSTMENT` с комментарием «Инвентаризация № X от ДД.ММ.ГГГГ» по каждой расходящейся строке.
7. Если общая сумма корректировок > порога — требуется подтверждение OWNER.

### 4.8 Аналитика владельца

**Актор:** Владелец Андрей, конец месяца.

1. Открывает «Отчёты → Выручка за период».
2. Указывает период «прошлый месяц», группировку «по услугам». Видит топ-5 услуг по выручке и количеству.
3. Переключает на «по мастерам» — видит выручку каждого мастера.
4. Открывает «Отчёты → Расход материалов».
5. Видит таблицу: SKU × месяц = количество и сумма списания. Может развернуть SKU и увидеть привязку к нарядам.
6. Открывает «Отчёты → Расхождения нормы». Видит мастеров с наибольшими отклонениями. Кликает на мастера → видит конкретные наряды и комментарии.

## 5. Функциональные требования

> Каждое требование имеет уникальный идентификатор `FR-X.Y.Z` для трассировки в user stories и тестах.

### 5.1 Идентификация и доступ

#### FR-5.1.1 Регистрация владельца (первичная инициализация)

При первом запуске системы создаётся единственный пользователь с ролью `OWNER`. Создание происходит:

- Либо через CLI-команду `pnpm setup:owner` (вводит email, телефон, имя, пароль).
- Либо через web-форму первичной настройки, доступную только если в БД нет ни одного OWNER.

После создания владельца форма недоступна. Повторное создание OWNER возможно только через CLI.

#### FR-5.1.2 Приглашение сотрудника

Владелец и менеджер могут пригласить сотрудника:

1. В админ-приложении «Сотрудники → Пригласить».
2. Заполняют: имя, телефон, email, роль (`MANAGER` или `MASTER`), филиалы доступа.
3. Система генерирует одноразовую пригласительную ссылку (TTL 72 часа) и отправляет её на email/SMS.
4. Сотрудник переходит по ссылке, проходит OTP-подтверждение телефона, задаёт пароль.
5. Аккаунт активируется, появляется в списке сотрудников.

Просроченные приглашения помечаются как `EXPIRED`; их можно перевыпустить.

#### FR-5.1.3 Аутентификация сотрудников

- Способ: телефон + OTP (SMS-код, 6 цифр, TTL 5 минут, лимит 3 попытки) ИЛИ email + пароль.
- Сессия: JWT access (15 минут) в памяти SPA + refresh (30 дней) в httpOnly secure cookie.
- 2FA TOTP (Google Authenticator/Yandex Key) — опционально для всех сотрудников, обязательно для OWNER.
- Lock-out: 5 неудачных попыток подряд → блокировка на 15 минут.

#### FR-5.1.4 Аутентификация клиента

Способы (любой по выбору клиента):

- Телефон + OTP (SMS-код).
- Яндекс ID (OAuth 2.0).
- Telegram Login Widget.

Гостевая запись (см. FR-5.5.10) использует одноразовый OTP без создания пароля.

#### FR-5.1.5 Восстановление доступа

- Сотрудники: кнопка «Забыли пароль?» → email с одноразовой ссылкой (TTL 30 минут) для установки нового пароля. Альтернативно — OTP по телефону.
- Клиенты: вход всегда по OTP/OAuth, восстановление не требуется (нет постоянного пароля).

#### FR-5.1.6 Управление ролями и доступами

- Только OWNER может изменять роль существующего сотрудника.
- Изменение роли логируется в audit log с указанием старого и нового значения, инициатора, времени.
- Деактивация сотрудника (soft): сотрудник теряет доступ, но не удаляется. Связанные с ним сущности (наряды, записи) сохраняются.
- Удаление сотрудника невозможно, если он связан с историческими данными (только деактивация).

#### FR-5.1.7 Аудит-лог действий

Все критические действия логируются в audit log:

- Логин (успешный/неуспешный), выход.
- Изменение ролей, активация/деактивация.
- Изменение прайса.
- Корректировки склада > порога.
- Удаление/анонимизация клиента.
- Закрытие наряда менеджером без мастера.
- Запросы на доступ к ПДн (152-ФЗ).

Записи в audit log не редактируются, не удаляются (хранятся минимум 1 год).

### 5.2 Каталог услуг

#### FR-5.2.1 Категории услуг

- Категории: справочник «название», «иконка», «порядок отображения», «активна».
- CRUD категорий доступен OWNER.
- При деактивации категории все услуги в ней становятся скрытыми для клиента, но остаются в БД.

#### FR-5.2.2 Карточка услуги

Поля:

- Название (обязательно, ≤ 200 символов).
- Описание (обязательно для клиентского отображения, поддержка markdown — bold, italic, списки).
- Категория (обязательно).
- Длительность (минуты, мин 15, шаг 15).
- Базовая цена (₽).
- Тип ценообразования: `FIXED` (одна цена) | `BY_BODY_TYPE` (зависит от типа кузова).
- Если `BY_BODY_TYPE` — таблица «тип кузова → цена».
- Связанные нормы расхода (см. FR-5.2.5).
- Активна (boolean).
- Порядок в категории.

#### FR-5.2.3 История изменения цен

Каждое изменение цены услуги логируется (старое/новое значение, дата, инициатор). Доступно владельцу как вкладка «История цен» в карточке услуги.

#### FR-5.2.4 Просмотр каталога клиентом

В client-приложении клиент видит:

- Сгруппированный по категориям список активных услуг.
- Название, краткое описание, длительность, цену (если `BY_BODY_TYPE` — диапазон «от X до Y»).
- Кнопку «Выбрать».
- Уже выбранные услуги в виде «корзины» сверху, с подсчётом длительности и суммы.

При выборе нескольких услуг отображается ИХ суммарная длительность и сумма.

#### FR-5.2.5 Нормы расхода материалов

- В карточке услуги — отдельная вкладка «Нормы расхода».
- Список строк: «SKU», «количество (в базовой ед. изм. SKU)», «коэффициент по типу кузова» (опционально, например 1.0 для седана, 1.3 для внедорожника).
- При создании наряда из услуги нормы автоматически копируются в наряд как «нормативный расход».

#### FR-5.2.6 Расчёт цены услуги для записи

Алгоритм:

1. Если услуга с `FIXED` ценообразованием — цена фиксированная.
2. Если `BY_BODY_TYPE` — берётся цена для типа кузова авто из записи. Если для данного типа цена не задана — используется базовая цена услуги, отображается предупреждение для менеджера.
3. Сумма по записи = сумма цен всех услуг.
4. Скидки/промокоды — Phase 2.

### 5.3 Учёт материалов

#### FR-5.3.1 Справочник SKU

Карточка SKU содержит:

- Артикул (внутренний, генерируется автоматически или задаётся вручную, уникален).
- Название.
- Группа (`Химия`, `Расходники`, `Инструмент`).
- Базовая ед. изм. (`мл`, `г`, `шт.`, `м`, `л`, `кг`).
- Упаковка (опционально): «Канистра 5 л» = 5000 мл, «Пачка 50 шт.» = 50 шт.
- Штрих-код (опционально).
- Минимальный остаток на филиал (порог уведомления).
- Срок годности учитывается? (boolean).
- Описание (markdown).
- Фото (опционально).
- Активен (boolean; при деактивации SKU не доступен для новых движений, но история сохраняется).

#### FR-5.3.2 Поставщики

Справочник поставщиков:

- Название (обязательно).
- ИНН/КПП (опционально).
- Контакты: телефон, email, адрес.
- Контактное лицо: ФИО, должность, телефон.
- Банковские реквизиты (опционально, для будущих интеграций).
- Комментарий.
- Активен (boolean).

#### FR-5.3.3 Партии

- Партия создаётся автоматически при приходе.
- Атрибуты: SKU, поставщик, дата прихода, дата выпуска (опционально), срок годности (опционально), количество поступившее, количество остаток, цена закупки за единицу.
- Партии — основа FEFO/FIFO списания.

#### FR-5.3.4 Единицы измерения и пересчёты

- Учёт ведётся в **базовой** ед. изм. SKU (например, мл).
- Приход возможен в **упаковочных** ед. (например, «канистра 5 л» = автоматический пересчёт в 5000 мл).
- Списание мастером — в базовых ед. (мл, г, шт.).
- В UI отображается удобная для пользователя единица (например, «Полироль X — остаток 2.5 канистры (12 500 мл)»).

#### FR-5.3.5 Приход (приходный ордер)

- Документ «Приход» содержит: поставщик, номер накладной поставщика, дата накладной, филиал-получатель, прикреплённый файл (PDF/фото накладной).
- Позиции: SKU, упаковка/базовая ед., количество, цена за единицу, срок годности (если SKU отслеживает).
- При проводке: создаётся партия на каждую позицию, создаётся `StockMovement` типа `RECEIPT`, обновляется средневзвешенная себестоимость:
  ```
  новая_себестоимость = (текущий_остаток × текущая_себестоимость + кол-во_прихода × цена_прихода)
                      / (текущий_остаток + кол-во_прихода)
  ```
- Приход можно отменить до 24 часов после проводки, если по партии не было других движений; иначе — только корректировка.

#### FR-5.3.6 Списание на наряд

- Автоматическое: при закрытии наряда система списывает материалы по фактическому расходу мастера.
- Ручное: менеджер может списать материалы вне наряда (например, на пробу). Требует обязательного комментария.
- Стратегия выбора партии:
  - SKU отслеживает срок годности → FEFO (First Expired First Out).
  - SKU не отслеживает → FIFO.
- Если на момент списания недостаточно остатка — операция блокируется с ошибкой; менеджер должен оформить приход или скорректировать факт расхода.

#### FR-5.3.7 Корректировка остатков

- Документ «Корректировка»: SKU, филиал, направление (+/−), количество, причина (обязательно текст), дата.
- Если сумма корректировки (количество × средневзвешенная себестоимость) > порога (по умолчанию 5 000 ₽) — требуется подтверждение OWNER (статус `PENDING_APPROVAL` → `APPROVED` / `REJECTED`).
- Корректировка «−» уменьшает партии по FIFO.
- Корректировка «+» создаёт «псевдо-партию» с null-поставщиком и нулевой ценой (видна как «Корректировка от ДД.ММ.ГГГГ»).

#### FR-5.3.8 Перемещение между филиалами

- Документ «Перемещение»: филиал-отправитель, филиал-получатель, позиции (SKU, кол-во).
- Создаются парные движения: `TRANSFER_OUT` на отправителе и `TRANSFER_IN` на получателе.
- Партии переносятся как есть (или по FIFO выбираются партии и создаются «дочерние» партии на получателе).

#### FR-5.3.9 Инвентаризация

- Документ «Инвентаризация»: филиал, дата начала, дата окончания, статус (`DRAFT`, `IN_PROGRESS`, `COMPLETED`).
- При создании в `IN_PROGRESS` фиксируется «системный остаток» на момент создания по каждой SKU.
- Менеджер по ходу инвентаризации вводит фактические количества.
- При проводке — создаются `StockMovement` типа `ADJUSTMENT` для расходящихся строк (одна корректировка на SKU, со ссылкой на инвентаризацию).
- Печать инвентаризационной ведомости (PDF) и сличительной ведомости (PDF) — обязательно для MVP.

#### FR-5.3.10 Минимальные остатки и уведомления

- В карточке SKU задаётся `reorderLevel` отдельно по каждому филиалу.
- При каждом списании система проверяет: текущий остаток < reorderLevel → создаётся уведомление «Низкий остаток» (FR-5.7.7).
- Уведомление дедуплицируется: одно уведомление на SKU+филиал не чаще раза в сутки.

#### FR-5.3.11 Штрихкоды и QR

- При вводе SKU в любом контексте (приход, корректировка, факт расхода) можно отсканировать штрихкод/QR — система найдёт SKU.
- Сканирование через камеру в браузере (BarcodeDetector API / fallback на ZXing).

#### FR-5.3.12 Себестоимость

- Хранится средневзвешенная себестоимость на уровне SKU (текущая) и на уровне партии (фиксированная по приходу).
- Для отчётов используется текущая средневзвешенная.
- Доступ к себестоимости — только OWNER.

### 5.4 Клиенты и автомобили

#### FR-5.4.1 Карточка клиента

- ФИО (обязательно).
- Телефон (обязательно, формат +7XXXXXXXXXX, уникален).
- Email (опционально).
- Дата рождения (опционально).
- Источник (опционально: `INSTAGRAM`, `REFERRAL`, `DIRECT`, `OTHER`).
- Согласие на обработку ПДн (обязательно, дата согласия фиксируется).
- Согласие на маркетинговые уведомления (опционально).
- Тип: `REGULAR` | `GUEST`.
- Комментарий (только для сотрудников).
- Дата создания, дата последнего посещения.

#### FR-5.4.2 Автомобили клиента

- Один клиент → много авто.
- Атрибуты: марка (обязательно), модель (обязательно), тип кузова (обязательно), госномер (опционально, формат А000АА00 или иной), VIN (опционально, 17 символов), год (опционально), цвет (опционально), комментарий.
- Авто принадлежит клиенту; нельзя «передать» авто другому клиенту, но можно деактивировать и создать новое.

#### FR-5.4.3 История посещений

В карточке клиента — лента: список записей и нарядов, фото «до/после», списанные материалы (без сумм для менеджера, с суммами для владельца).

#### FR-5.4.4 Удаление и анонимизация (152-ФЗ)

- Клиент в client-приложении может запросить удаление аккаунта.
- Запрос обрабатывается OWNER в течение 30 дней.
- При удалении: ФИО, телефон, email, авто-данные клиента анонимизируются (заменяются на `ANONYMIZED_<id>`); связанные наряды/записи остаются с этой пометкой; ПДн в audit log также маскируются.
- Полное удаление невозможно (нужны исторические записи для бухгалтерии).

#### FR-5.4.5 Поиск и фильтрация клиентов

В админ-приложении список клиентов с фильтрами:

- По ФИО (substring, case-insensitive).
- По телефону (включая частичный).
- По госномеру авто.
- По дате последнего посещения (диапазон).
- По типу (`REGULAR`/`GUEST`).
- По наличию активных записей.

Сортировка: по имени, по дате создания, по дате последнего посещения.

### 5.5 Онлайн-запись и расписание

#### FR-5.5.1 График работы филиала

- Для каждого филиала: расписание по дням недели (ПН–ВС).
- Часы открытия и закрытия (например, 09:00–21:00).
- Перерыв (опционально).
- Исключения: праздники, нерабочие дни, особые часы (например, 31 декабря — короткий день).
- В нерабочие часы запись недоступна.

#### FR-5.5.2 График мастеров

- Для каждого мастера: расписание по дням недели (может быть «через день», «2/2», свой режим).
- Часы (могут отличаться от филиала, но не выходить за рамки филиала).
- Отпуска и отгулы (диапазон дат, в которые мастер недоступен).

#### FR-5.5.3 Расчёт доступных слотов

Алгоритм для пары (филиал, услуги, авто-тип кузова, мастер опциональный):

1. Длительность = сумма длительностей услуг.
2. Период поиска: ближайшие 14 дней (настраиваемо).
3. Шаг сетки: 15 минут.
4. Для каждого мастера в филиале (или для конкретного мастера):
   - Берём график мастера на каждый день, исключая отпуска и закрытые рабочие часы.
   - Вычитаем существующие записи (с буфером 0 минут — настраиваемо).
   - Получаем «окна» свободного времени.
   - В каждом окне выделяем слоты длительностью = длительности услуг с шагом 15 минут.
5. Объединяем слоты по всем мастерам (если выбран «любой свободный»).
6. Возвращаем слоты, отсортированные по времени.

#### FR-5.5.4 Просмотр доступных слотов клиентом

- Календарь с переключением по дням.
- Каждый слот: время начала, длительность, мастер (если выбран), свободные/занятые.
- Только свободные слоты кликабельны.
- На уровне дня — индикатор «есть слоты / нет слотов».

#### FR-5.5.5 Создание записи клиентом (онлайн)

Происходит автоматическое подтверждение при выполнении условий:

- Клиент авторизован (OTP, OAuth) — да.
- Указанные услуги активны — да.
- Слот свободен — да (повторная проверка в момент создания, защита от race-condition через оптимистичную блокировку).
- Тип кузова авто соответствует прайсу услуг — да.

При выполнении — статус `CONFIRMED`. Если что-то не выполнено — `PENDING_CONFIRMATION` с уведомлением менеджера.

#### FR-5.5.6 Создание записи менеджером

Менеджер из админ-приложения может:

- Создать запись на существующего или нового клиента.
- Выбрать любой слот (включая занятые — с предупреждением и подтверждением).
- Изменить цену по позиции (с указанием причины — скидка, специальная цена) — только если OWNER разрешил `MANAGER` менять цены в карточке записи (по умолчанию: запрещено).

Запись от менеджера сразу `CONFIRMED`.

#### FR-5.5.7 Отмена и перенос записи клиентом

Правила:

- ≥ 24 часа до начала — самостоятельно (статус → `CANCELLED`, слот освобождается).
- < 24 часа — заявка `CancellationRequest` со статусом `PENDING`. Менеджер `APPROVE` или `REJECT`.
- Если запись уже `IN_PROGRESS` — отмена клиентом запрещена.

#### FR-5.5.8 Расписание мастера в master-приложении

- Лента записей на сегодня и завтра.
- Карточка записи: время, клиент, авто, услуги, статус.
- Действия: «Принять авто» (`CONFIRMED` → `IN_PROGRESS` + создание наряда), «Закрыть наряд», «Связаться с клиентом» (телефон).

#### FR-5.5.9 Расписание в админ-приложении

- Дневной и недельный вид.
- Сетка: ось X — время, ось Y — мастера (или боксы — переключение).
- Drag & drop переноса записи на другого мастера или другое время (с подтверждением).
- Цветовая индикация статусов.

#### FR-5.5.10 Гостевая запись

- Не требует регистрации.
- Минимальный набор данных: ФИО, телефон, тип авто.
- OTP по SMS — обязательно.
- В системе создаётся «лёгкая» карточка клиента типа `GUEST`.
- После записи — предложение перейти в полноценный аккаунт (создать пароль или подключить OAuth).

### 5.6 Наряды (выполнение услуги)

#### FR-5.6.1 Создание наряда

- Создаётся автоматически при переходе записи в `IN_PROGRESS`.
- Один наряд на одну запись.
- Атрибуты копируются из записи: клиент, авто, мастер, услуги, нормы расхода.
- Состояние: `OPEN`.

#### FR-5.6.2 Фото «до/после»

- Загрузка фото через камеру (мобильный) или загрузку файла (десктоп).
- Хранение в MinIO/S3.
- Минимум 1 фото «до» и 1 фото «после» обязательно для закрытия наряда.
- Максимум 20 фото каждого типа (мягкий лимит).
- Размер файла ≤ 10 МБ, форматы JPEG, PNG, HEIC (с конверсией).
- Сжатие на клиенте до 1920px по большей стороне.

#### FR-5.6.3 Состояния наряда

```
OPEN           → IN_PROGRESS  → AWAITING_REVIEW → CLOSED
                                                ↘
                                                  CANCELLED
```

- `OPEN`: создан, но мастер ещё не начал.
- `IN_PROGRESS`: мастер фактически работает; ставится при добавлении первой строки факта расхода или загрузке первого фото «до».
- `AWAITING_REVIEW`: мастер указал, что закончил, но требуется ревью (опционально, по настройке студии).
- `CLOSED`: наряд завершён, материалы списаны, клиент уведомлён.
- `CANCELLED`: отмена (например, авто увезли); требует комментария.

#### FR-5.6.4 Фактический расход материалов

- Мастер добавляет строки: SKU, количество (в базовых ед.).
- При добавлении SKU система подсказывает норму (если есть для услуги наряда), показывает текущий остаток в филиале.
- Можно сканировать штрихкод SKU.
- Для каждой строки можно указать комментарий.
- При закрытии наряда: для каждой строки рассчитывается отклонение от нормы. Если |отклонение / норма| > 15% — мастер обязан указать причину; без причины закрытие невозможно.

#### FR-5.6.5 Закрытие наряда

Условия:

1. ≥ 1 фото «до» загружено.
2. ≥ 1 фото «после» загружено.
3. Все строки факта расхода имеют корректные значения и (если требуется) причину расхождения.
4. Все услуги отмечены как выполненные (опционально, можно не отмечать — все услуги наряда считаются выполненными по умолчанию).

При закрытии:

- Создаются `StockMovement` типа `CONSUMPTION_WRITE_OFF` по каждой строке расхода (партия выбирается по FEFO/FIFO).
- Запись переходит в `COMPLETED`.
- Триггерится уведомление клиенту (FR-5.7.6).
- Audit log: «Наряд закрыт мастером ID за время X».

#### FR-5.6.6 Корректировка наряда после закрытия

- Только OWNER или MANAGER может открыть закрытый наряд для корректировки.
- Корректировка факта расхода → создаются обратные движения (отмена старого списания) + новые движения с обновлёнными количествами.
- Все корректировки логируются с указанием инициатора и причины.

#### FR-5.6.7 Отмена наряда

- Менеджер/владелец может отменить наряд из любого статуса до `CLOSED`.
- При отмене материалы НЕ списываются; если уже были предварительно зарезервированы — резерв снимается.
- Указывается причина отмены (обязательно).

Все шаблоны на русском, параметризованы Handlebars-подобной нотацией `{{name}}`, `{{datetime}}` и т.п.

Список шаблонов MVP:

| Код | Адресат | Триггер | Канал по умолчанию |
|---|---|---|---|
| `APPOINTMENT_CONFIRMED` | клиент | запись подтверждена | SMS, Telegram, Push |
| `APPOINTMENT_REMINDER` | клиент | за 24 ч / за 2 ч до начала | SMS, Telegram, Push |
| `APPOINTMENT_CANCELLED` | клиент | запись отменена | SMS, Telegram, Push, Email |
| `APPOINTMENT_RESCHEDULED` | клиент | запись перенесена | SMS, Telegram, Push, Email |
| `WORK_ORDER_COMPLETED` | клиент | наряд закрыт | SMS, Telegram, Push, Email (с фото) |
| `CANCELLATION_REQUEST_NEW` | менеджер | новый запрос на отмену внутри окна | Telegram, Push, Email |
| `LOW_STOCK` | владелец, менеджер | остаток ниже минимума | Telegram, Push, Email |
| `OWNER_DIGEST_DAILY` | владелец | ежедневная сводка (выручка, число записей, низкие остатки) | Telegram, Email |

#### FR-5.7.3 Подтверждение записи

После создания записи (онлайн или менеджером) клиенту отправляется уведомление по каналам, на которые он подписан. По умолчанию для гостя — SMS.

#### FR-5.7.4 Напоминание о записи

- За 24 часа: одно напоминание.
- За 2 часа: одно напоминание (опционально включается студией).
- Напоминание в Telegram содержит инлайн-кнопку «Подтвердить» / «Отменить».

#### FR-5.7.5 Уведомление менеджера о новой/изменённой записи

- Канал: Telegram (групповой чат студии) + email фолбэк.
- Содержание: ссылка на карточку записи.

#### FR-5.7.6 Уведомление о закрытии наряда

- Клиент получает уведомление с фото «до/после» и списком выполненных услуг.
- Email — с прикреплёнными изображениями (или ссылкой на личный кабинет, если суммарный размер > 5 МБ).
- SMS/Telegram — текст + ссылка на личный кабинет.

#### FR-5.7.7 Уведомление о низком остатке

- Триггер: списание привело к остатку ниже минимума.
- Дедупликация: не чаще раза в 24 часа на SKU+филиал.
- Получатели: владелец и менеджеры, имеющие доступ к филиалу.

#### FR-5.7.8 Настройки уведомлений пользователем

- В личном кабинете пользователя — настройка по каждому шаблону: каналы (вкл/выкл).
- Глобальный «не беспокоить» — диапазон часов, в которые уведомления откладываются (кроме критических, например, изменение/отмена записи).
- Отписка по каждому каналу: ссылка `/unsubscribe?token=...` в email/SMS.

#### FR-5.7.9 Очередь и доставка

- Уведомления отправляются асинхронно через BullMQ-очередь.
- Retry: 3 попытки с экспоненциальной задержкой.
- Failed-уведомления попадают в DLQ, доступны OWNER в техническом разделе админки.

### 5.8 Биллинг подписки (демо-заглушка)

#### FR-5.8.1 Тарифы (демо)

В системе предзаполнены 3 тарифа:

| Код | Название | Цена/мес | Лимит филиалов | Лимит мастеров | Лимит записей/мес |
|---|---|---|---|---|---|
| `STARTER` | Стартер | 2 990 ₽ | 1 | 3 | 200 |
| `STANDARD` | Стандарт | 5 990 ₽ | 3 | 10 | 1 000 |
| `PRO` | Про | 11 990 ₽ | ∞ | ∞ | ∞ |

#### FR-5.8.2 Личный кабинет тарифа

- В админ-приложении OWNER видит: текущий тариф, дату следующего списания, баланс.
- Может «сменить тариф» (демо: смена применяется мгновенно, без реальной оплаты).
- Видит историю списаний (демо: записи генерируются автоматически каждый месяц).

#### FR-5.8.3 Имитация оплаты

- Кнопка «Оплатить» открывает модалку «Подключить ЮKassa» с заглушкой «В разработке».
- Реальная интеграция с ЮKassa Recurring — Phase 2.

#### FR-5.8.4 Лимиты тарифов

- В MVP лимиты не enforce-ятся (это демо). При превышении показывается предупреждение «Превышен лимит тарифа `Стартер`. Пожалуйста, перейдите на `Стандарт`».

## 7. Нефункциональные требования

### 7.1 Производительность

| Метрика | Цель MVP |
|---|---|
| API latency P50 (читающие) | ≤ 100 ms |
| API latency P95 (читающие) | ≤ 300 ms |
| API latency P99 (читающие) | ≤ 800 ms |
| API latency P95 (записывающие) | ≤ 600 ms |
| API latency P95 (отчёты) | ≤ 2 s |
| Frontend FCP (4G, средний девайс) | ≤ 1.8 s |
| Frontend LCP | ≤ 2.5 s |
| Frontend TTI | ≤ 3.5 s |
| Bundle size (initial) admin | ≤ 350 КБ gzip |
| Bundle size (initial) client | ≤ 200 КБ gzip |
| Bundle size (initial) master | ≤ 250 КБ gzip |

### 7.2 Надёжность

- Uptime: 99% (best-effort на MVP, без формального SLA).
- RTO (Recovery Time Objective): 4 часа.
- RPO (Recovery Point Objective): 24 часа (ежедневный бэкап).
- Бэкапы: ежедневный pg_dump в MinIO/S3, retention 30 дней.

### 7.3 Безопасность

- TLS 1.2+ обязателен для всех публичных endpoints.
- Хранение паролей: bcrypt, cost factor 12.
- Хранение JWT-секретов: env, в production — Vault/Secret Manager (Phase 2).
- CSRF-защита: SameSite=Strict для refresh-cookie + double-submit token для не-GET мутаций (если используются куки в сессии).
- CORS: явный allowlist origins.
- Helmet: CSP, X-Frame-Options, X-Content-Type-Options.
- Rate limiting: 60 req/min на IP для публичных endpoint, 600 req/min для авторизованных, 5 OTP-запросов в час на телефон.
- Валидация всех входов: zod / class-validator на DTO.
- Sanitization вывода: автоматическое экранирование Angular для XSS.
- SQL-инъекции: только параметризованные запросы через ORM, никакой ручной конкатенации.
- Шифрование at-rest: ПДн (телефон, email, ФИО) — pgcrypto column-level encryption (опционально на MVP, обязательно на Phase 2).
- 2FA: опциональное для всех, обязательное для OWNER.
- Аудит-лог критических действий (см. FR-5.1.7).

### 7.4 Соответствие 152-ФЗ

- Хранение ПДн на серверах в РФ (Yandex Cloud, Selectel, VK Cloud).
- Явное согласие на обработку при регистрации (отдельный чекбокс, нельзя предзаполнен).
- Отдельное согласие на маркетинговые рассылки.
- Право на доступ: клиент может выгрузить свои ПДн в JSON.
- Право на удаление: клиент может запросить удаление; OWNER обрабатывает в течение 30 дней.
- Анонимизация вместо удаления для исторических данных.
- Журнал доступа к ПДн: фиксируется любой запрос ПДн «третьего лица» (клиента просматривает менеджер/мастер) с указанием инициатора.
- Документ «Политика обработки ПДн» — публикуется на client-приложении, ссылка обязательна на форме регистрации.

### 7.5 Локализация

- MVP: только русский язык.
- Архитектурно поддерживаются другие локали через `@ngx-translate/core` или Angular i18n.
- Все строки UI вынесены в JSON-словари.
- Даты: формат DD.MM.YYYY, время — HH:mm.
- Валюта: RUB, формат «1 234 ₽» (пробел как разделитель тысяч).
- Часовой пояс: задаётся на уровне филиала из списка IANA TZ; UI отображает в локальном TZ филиала.

### 7.6 PWA и offline

- Все три SPA — PWA (manifest, service worker).
- Установка на главный экран мобильного устройства.
- Offline-режим:
  - **client**: только просмотр своих записей (read-only кэш).
  - **master**: полный offline для существующих нарядов (фото добавляются в очередь, синхронизируются при появлении сети). Это критично для боксов с плохой связью.
  - **admin**: read-only кэш справочников.
- Web Push уведомления через VAPID.

### 7.7 Доступность (a11y)

- Целевой уровень: WCAG 2.1 AA.
- Контраст текста ≥ 4.5:1.
- Все интерактивные элементы доступны с клавиатуры.
- Корректные ARIA-роли в кастомных компонентах (или используем готовые из Taiga UI с встроенной a11y).
- Скринридер-friendly формы (label-for, aria-describedby для ошибок).
- Альт-тексты для всех изображений.
- Размер кликабельной области ≥ 44×44 px.

### 7.8 Логирование и наблюдаемость

- Логи: structured JSON через Pino.
- Уровни: `error`, `warn`, `info`, `debug` (в production — info+).
- Correlation ID: каждый HTTP-запрос получает X-Request-Id, пробрасывается через все логи и доменные события.
- PII в логах: маскируется (телефон → +7***1234, email → e***@example.com).
- Health-checks: `/health/live` (всегда 200, если процесс жив), `/health/ready` (200 если БД и Redis доступны).

### 7.9 Бэкапы и восстановление

- pg_dump ежедневно в 03:00 (по TZ деплоя), хранится в MinIO 30 дней.
- MinIO bucket с фотографиями — реплицируется на резервный bucket еженедельно (Phase 2).
- Скрипт восстановления `restore.sh` документирован в README.
- Регулярные drill-восстановления (минимум раз в квартал) — Phase 2.

### 7.10 Совместимость браузеров

- Chrome/Edge ≥ 110.
- Safari ≥ 16 (включая iOS 16+).
- Firefox ≥ 110.
- Не поддерживается: IE, Opera Mini.

### 7.11 Объём данных (расчётная нагрузка MVP)

- Студия среднего размера: ~ 50 записей/день × 30 дней = 1 500 записей/месяц.
- 1 наряд = 5–15 строк факта расхода = 7 500–22 500 движений склада/месяц.
- Фото: 10 фото на наряд × 1 МБ = 10 МБ × 1 500 нарядов = 15 ГБ/месяц медиа.
- БД: ~ 100 МБ/месяц для основных таблиц, ~ 1 ГБ/год за вычетом аудит-логов.
- Расчётный пик одновременных пользователей: 5 (мастера + админ).

## 8. UI/UX и информационная архитектура

### 8.1 Принципы дизайна

- **Простота важнее красоты.** Студия — это рабочий инструмент мастеров и админа; интерфейс должен быть быстрым и предсказуемым.
- **Минимум кликов до действия.** Ключевые сценарии (открыть наряд, добавить расход, найти запись) — не более 3 кликов от стартового экрана.
- **Mobile-first для master и client.** Master работает на планшете в боксе, client — на телефоне.
- **Desktop-first для admin.** Админ работает за компьютером с большим экраном.
- **Тёмная тема.** Поддерживается во всех приложениях, переключение в профиле пользователя.

### 8.2 Дизайн-токены

База — **Taiga UI** + **Tailwind CSS**.

- Цветовая палитра:
  - Primary: Taiga `tui-primary` (настраивается на брендинг студии).
  - Accent: индивидуальный «детейлинг»-зелёный/синий.
  - Status: success/warning/error из Taiga.
- Типографика: Taiga шрифт `Manrope` (или Inter); размеры по Taiga.
- Spacing: Tailwind 4px scale (`p-2`, `p-4`, `p-6`, `p-8`).
- Радиусы: Taiga `m`, `l`.
- Тени: Taiga `tui-shadow`.

> Taiga UI задаёт глобальные CSS-переменные для темы; Tailwind не должен ломать их через произвольные `bg-*`/`text-*` без согласования. Tailwind используется в первую очередь для **layout** (`flex`, `grid`, `gap`, `p-*`, `m-*`), а не для цветов.
